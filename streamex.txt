C2R Stream Explorer
C2R Stream Explorer is a tool to view and extract the contents of Office C2R's stream*.dat files.

Usage
Use File/Open to specify an input directory. Stream Explorer will enumerate the contents of the directory and open all the streams it will find inside it (a C2R build typically contains several streams).

The File Explorer view will contain a directory tree displaying the contents from all the streams found in the input directory.

Expanding a directory tree view item will display the contents of that directory, if the directory contains any files, the files will be listed in a list view the right pane, displaying name, version (if versioned) and size.

Right clicking on a file displays a single command, Extract, clicking it will extract the file to the specified extract directory (by default, the current directory).

Use Options/Extract Preferences to specify a folder to extract files to.

Stream Explorer can open C2R builds (releases as well as install root images) as well as the outputs of devc2r or relc2r target roots.

Command line syntax
When run from the command line, it's possible to display usage and optionally specify the input directory as well as the extract directory:


StreamExplorer.exe /?

Usage:
StreamExplorer.exe <streamsDirectory>
Will open all the streams in the specified <streamsDirectory> (at any depth).

StreamExplorer.exe <streamsDirectory> <extractDirectory>
Will open all the streams in the specified <streamsDirectory> (at any depth) and
use <extractDirectory> when extracting files from the stream(s).

StreamExplorer.exe /?
Will display this message.
Implementation/Internals
C2R Stream Explorer is a Windows Presentation Foundation (WPF) application, written in C#.

Stream parsing
Stream content parsing is done via ManagedStreamShared, a C++/CLI (Common Language Infrastructure) DLL built by the click2run project (click2run/Stream/Shared/ManagedDLl), which provides a managed code interface to the underlying C++ implementation of the native C2R Stream code.

Sources
C2R Stream Explorer sources are in click2runtools\StreamExplorer.

Build and test
Build using ohome
When making changes, create a private branch off the official/c2rclient branch.

Building the click2runtools project (for x64 platform) will generate a StreamExplorer.exe (and .dll etc.) under %TARGETROOT%\x64\<flavor>\click2runtools\x-none.

If running StreamExplorer directly from the click2runtools target root, its dependencies need to be first copied next to it, e.g. (substitute flavor as appropriate):


copy %SRCROOT%\otools\bin\Microsoft.Deployment.Compression.* %TARGETROOT%\x64\ship\click2runtools\x-none
copy %TARGETROOT%\x64\ship\click2run\x-none\ManagedStreamShared.* %TARGETROOT%\x64\ship\click2runtools\x-none
Building in Visual Studio
Open the solution using 'devdiv %SRCROOT%\click2runtools\StreamExplorer\StreamExplorer.sln' from the Office command prompt.
StreamExplorer can be built, debugged, run from etc. inside Visual Studio.

Known issues/Roadmap
Registry Explorer view is empty (should be removed if un-implemented or fixed if it is supposed to be working).

Package the tool (with its dependencies) inside a NuGet package for easier downstream consumption.

Add filtering/search for files, etc.

Allow selection of subset of streams to be displayed (once registry explorer view is added, consider same for .reg files)

Add an 'MCXML Explorer' tab.

Add command line support for ProjFS provider functionality.

Add 'extract' support at the directory level

Add support (in both UI and from command line) for extracting multiple files, single or multiple directories, single stream or multiple streams

Currently all manifests are 'merged/added' together when processing the input directory and then enumerated on the fly as the stream 'tree view' nodes get expanded in the File Explorer view. In order to support item 4 (and 6, 8) above, consider changing this architecture to have the enumeration/extraction happen a stream at a time (ideally, asynchronously but to start with synchronously if perf is not too bad), annotating the backing store with which specific stream each file comes from.
