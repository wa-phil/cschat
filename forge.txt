Overview of Forge
Forge is the main tool used to orchestrate the C2R build process. A typical C2R build will execute several Forge processes, each performing a specific action. Where possible these Forge processes will run concurrently and where required they are run sequentially. A typical Forge process will consume as inputs the outputs of a previous invocation, process them in some way and in turn produce outputs to be used as inputs of the next process and so on. There are currently over 30 actions defined, please see List of Forge actions for details.

Forge telemetry
Forge is instrumented and logs telemetry to enable us to monitor the evolution of its performance over time, to diagnose regressions, etc. See Forge dashboard link at Deploy tools dashboards.

To make changes to Forge in the current branch you must enlist in a private branch (e.g. user/<alias>{=html}/<work item #>) off of the official/deploytools branch in OMR.

Forge source code is located under deploy\forge for the executable and deploy\c2rcommon for shared libraries.

The following steps can be applied to any of the Nuget packages built by pkgnuget_deploy, just replace Forge with Packages in the script name and add <Package>{=html} as initial argument (e.g. instead oftestCBForge ``<args>{=html} usetestCBPackages ``<package>{=html} ``<args>{=html}), where<Package>{=html} is the appropriate Nuget package name fromBinaryStore|Devinstall|Gimmetools.

When calling the generic versions of the scripts (preparePackagesForLocal, resetLocalPackages, testCBPackages, uploadPackages), if no package is specified then the script will iterate over all valid/applicable packages (for example, omitting Devinstall for testCBPackages since currently there is no affordance to test DevInstall<C2R|Cent> in CloudBuild.

Testing in CloudBuild
When you get confidence that your change is ready to be checked in you can always test the changes in CloudBuild. To do so you will need to follow this step:

Run the following command to build a test Forge Nuget package, publish it, update the .packageset references to it, package the specified changelist # (or all pending files) as DPK and trigger a CloudBuild buddy build with the changes:
powershell %SRCROOT%\pkgnuget\deploy\Scripts\testCBForge.ps1 -ChangeList <cl#|all>

Please note that if the above script fails with a permission related error then one has to request permissions, here is the simple 10-step process:

Go to https://aka.ms/myaccess
Click in the search box at the top of the page
Search for "OV_Pkg_Office_Feed"
Click on the OV_Pkg_Office_Feed link
Click Request Membership / For Myself
Select the Exemption† radio button under Role
Enter business justification (e.g. "Need to test updated Forge package in CB before submitting a change.")
Review the request, if you need to make changes click Make Changes, if not, proceed to next step
Click Submit Request
This is it, you should receive emails keeping you abreast of the progress and once access is granted then you should be able to run the PowerShell command line above successfully.

† Exemption: Grants temporary access to publish packages to Office feed. If you are publishing any packages using your identity, then please do consider ADO pipeline to publish official packages. Soon we will be restricting individuals to publish packages to Official feed

RepoMan considerations
RepoMan is the main toolset being used for the C2RX/Unity effort. It is used both at build time (by Forge) to create C2R Unity packages as well as being included in the C2R builds to be used at install time (by C2R Client).

RepoMan is developed outside of the Office Source Depot, in a Git repository, updates to RepoMan are picked up when the RepoMan packageset is updated to refer to a new build version of RepoMan.

At build time, Forge both uses and packages RepoMan from the version referenced in the RepoMan packageset (otools\deps\pkgsets\common\microsoft.office.c2runity.repoman.packageset).

Ideally, RepoMan updates (i.e. changes to the version in the RepoMan packageset) should be made so they're compatible with current versions of both Forge and C2RClient.

In cases where breaking changes have to be made, the solution is to check-in the RepoMan packageset change at the same time as the matching edits to either Forge or C2RClient.

Since edits to Forge are made in the deploytools override and edits to C2RClient are (at least usually) in the tenantc2rclient override, this would require adding the RepoMan packageset to the override(s), see earlier in this page for steps.

An added complication, in the case of Forge, is that Forge itself is consumed from a Nuget package (otools\deps\pkgsets\common\microsoft.tools.office.c2r.forge.packageset) and this packageset file is normally updated automatically when changes are checked-in to the override and build successfully, as per above.
