## Project Overview

CSChat is an interactive C# console/GUI chat client for LLM servers (Ollama and Azure OpenAI). It supports RAG, autonomous planning, a tool system, MCP (Model Context Protocol), and optional subsystems like Azure DevOps, Kusto, and email integration.

## Common Commands

```bash
# Build
dotnet build

# Run
dotnet run
dotnet run -- -h http://localhost:11434 -m <model-name>
dotnet run -- -p azureai -u gui

# Run tests
dotnet test

# Run a specific test
dotnet test --filter "DisplayName~JSONParser"
```

**Command-line flags:** `-h` (host), `-m` (model), `-s` (system prompt), `-p` (provider: `ollama`|`azureai`), `-e` (embedding model), `-u` (UI: `terminal`|`gui`)

## Architecture

### Core Files

| File | Purpose |
|------|---------|
| `Program.cs` | Entry point, DI container setup, main chat loop |
| `Engine.cs` | Chat orchestration, RAG store management, file ingestion |
| `Context.cs` | Chat message history and system prompt management |
| `Config.cs` | Configuration model, `config.json` persistence |
| `Commands.cs` | `Command` class, `CommandManager`, menu navigation |
| `Interfaces.cs` | Core interfaces: `IChatProvider`, `ITool`, `ISubsystem`, `IVectorStore`, `ITextChunker`, `IUi` |
| `Log.cs` | Context-aware structured logging with retry support |
| `Graph.cs` | Knowledge graph for advanced RAG |

### Directory Structure

- **`/Providers/`** ‚Äî `IChatProvider` implementations (Ollama, AzureAI)
- **`/Tools/`** ‚Äî `ITool` implementations and `ToolRegistry`; `Planner.cs` for autonomous multi-step execution
- **`/Memory/`** ‚Äî `VectorStore.cs` (in-memory, with MMR support)
- **`/Chat/`** ‚Äî Persistent chat thread management (`ChatManager`, `ChatThread`)
- **`/Commands/`** ‚Äî Command implementations grouped by domain (RAG, providers, system, subsystems)
- **`/Subsytems/`** ‚Äî Optional plugin subsystems (ADO, Kusto, MAPI, MCP, PRs, S360)
- **`/UX/`** ‚Äî `IUi` abstraction with `Terminal.cs` and `Photino.cs` (web GUI) implementations
- **`/Json/`** ‚Äî Custom lightweight JSON parser/writer (used for config, not general-purpose)
- **`/UserManagedData/`** ‚Äî Reflection-based CRUD system for user-defined data collections
- **`/unittests/`** ‚Äî xUnit test project

### Key Architectural Patterns

**Dependency Injection:** `Microsoft.Extensions.DependencyInjection` wires providers, tools, chunkers, and subsystems at startup.

**Command Pattern:** All interactive commands are instances of the `Command` class. Commands are registered in `CommandManager` and navigate via the menu system (press Escape in terminal mode).

**Subsystem Plugin Architecture:** `ISubsystem` implementations are discovered via reflection on `[IsConfigurable]` attributes. `SubsystemManager` handles lifecycle (connect/disconnect) and config persistence. Adding a new subsystem means implementing `ISubsystem` in `/Subsytems/`.

**User-Managed Data:** Classes marked `[UserManaged]` with `[UserField]`/`[UserKey]` properties automatically get CRUD commands and persist in `config.json`. Used for chat threads, saved queries, etc.

**Tool System:** `ITool` provides name/description/schema. Tools are registered in `ToolRegistry` and can be invoked by the model or by the `Planner` for autonomous execution (max 25 steps default).

**Logging Pattern:**
```csharp
await Log.MethodAsync(async ctx => {
    ctx.OnlyEmitOnFailure();
    // work here
    ctx.Succeeded();
});
```

### Configuration

Settings persist to `config.json` (alongside the executable). Key settings: provider, model, host, temperature, max tokens, system prompt, RAG settings, subsystem enable/disable flags.

### Build-Generated Files

`Generated/BuildInfo.cs` is auto-generated during pre-build via MSBuild using `git describe --tags` and `git rev-parse`. Do not edit this file manually; edit `BuildInfo.template.cs` instead.

## Testing
